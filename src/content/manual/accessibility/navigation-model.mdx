The application uses a **tab-group** navigation model (the WAI-ARIA composite-widget / roving-tabindex pattern). Interactive controls are organised into groups; <Shortcut client:load keys="tab" /> moves between groups, arrow keys move within a group.

Two hooks implement this pattern:

- **Container-level roving** (`useContainerTabGroup`) — one physical element in the group carries the tab stop. Used by the transport and tool toolbars, the selection toolbar, the effects panel, and clip cycling inside a track.
- **Per-item roving** (`useTabGroup`) — each item is its own DOM node, and the tab stop moves between them. Used by the project toolbar's tabs, the effects panel's grid cells, and dialog field groups (Export, Preferences).

Both follow the same rules below; two rules apply to only one of the two hooks, called out in the table.

## Core rules

| Rule | Behaviour |
|---|---|
| One tab stop per group | Exactly one element in each group carries the group's roving `tabIndex`; every other element in the group has `tabIndex = -1` |
| <Shortcut client:load keys="tab" /> / <Shortcut client:load keys="shift+tab" /> | Move forward/backward **between groups** |
| Arrow keys | Move focus **within** the current group — Right/Down to the next item, Left/Up to the previous — wrapping from last to first when the group's configuration enables `wrap` |
| <Shortcut client:load keys="home" /> / <Shortcut client:load keys="end" /> | Jump to the first / last item in the group |
| Hidden items | Skipped during arrow navigation (container-level groups also skip disabled items) |
| Blur reset — *container-level groups only* | When focus leaves a `useContainerTabGroup` group, its first visible element becomes the tab stop again — re-entering the group always resumes at the first item, never wherever focus last left it |
| Focus-on-entry reset — *per-item groups only* | When focus enters a `useTabGroup` group from outside, it always lands on the first item, regardless of which item last held the tab stop |
| Unknown group id — *container-level groups* | If a group id isn't in the active profile's configuration map, `useContainerTabGroup` falls back to roving + arrows + wrap rather than failing |

## Global tab order

Groups are visited in this order (numbers are the roving `tabIndex` values assigned by the AU4 Tab Groups profile):

| # | Group | tabIndex |
|---|---|---|
| 1 | File menu (menubar) | 1 |
| 2 | Project toolbar | 2 |
| 3 | Project toolbar — actions | 3 |
| 4 | Project toolbar — workspace | 4 |
| 5 | Project toolbar — history | 5 |
| 6 | Tool toolbar | 6 |
| 7 | Effects panel | 7 |
| 8 | Add new track | 98 |
| 9 | Timeline ruler | 99 |
| 10 | Track area (see below) | 100 and up |
| 11 | Selection toolbar (timecodes) | configured as 200 — see note below |

<Callout type="info" title="The file menu isn't always there">
The menubar (tabIndex 1) only exists in the **Windows** chrome variant, and only outside the Electron desktop shell. The macOS variant renders no menu bar at all (just the traffic lights and app name), and the Electron build suppresses the menubar in favour of the native OS menu. The rest of this table is unaffected either way.
</Callout>

<Callout type="warning" title="Selection toolbar's tabIndex 200 doesn't actually apply">
The selection toolbar is configured for tabIndex 200 in the profile, but its focusable elements are the `TimeCode` groups, and `TimeCode` hardcodes its own container to `tabIndex={0}` (and re-asserts `0` after menu interactions). Native Tab order visits every positive-tabIndex element first, then all `tabIndex=0` elements in DOM order — so the selection toolbar is actually observed at the very end of the sequence, not at position 200, and Tab from there wraps back to the file menu.
</Callout>

## Track area tab order

Each track contributes tab stops at a **stride of 4**, based off `trackBase = 100` (the `tracks` group's tabIndex). For track index `i` (0-based):

| Element | tabIndex |
|---|---|
| Track container (`.track`) | `100 + i × 4` |
| *(reserved, unused)* | `101 + i × 4` |
| Clips (first clip on the track; the rest are `-1`) | `102 + i × 4` |
| Per-track vertical ruler | `103 + i × 4` |

For a project with four tracks the live values are: containers `100, 104, 108, 112`; clips `102, 106, 110, 114`; rulers `103, 107, 111, 115`.

<Callout type="info" title="The track control panel isn't in this sequence">
The side panel of track controls (mute/solo, name, pan, volume) has `tabIndex={-1}` — it is deliberately **not** a native Tab stop. It's still reachable: the track container's own Tab handler moves focus into it programmatically. Treat it as attached to its track rather than as its own stop in the table above.
</Callout>

<Callout type="info" title="Debug profile: WCAG flat navigation">
The prototype also ships a debug profile (`wcag-flat`) in which every interactive element becomes its own tab stop (`tabIndex=0`, DOM order) and arrow-key grouping is turned off. It exists for comparison testing via the Debug Panel and is <strong>not product behaviour</strong> — the rest of this page documents the default AU4 Tab Groups profile only.

It does <strong>not</strong> disable clip editing keyboard shortcuts. A `keyboardShortcuts` gating block (covering clip moves and edge extend/reduce, and label move/trim/delete) exists in both profiles' configuration, but the only code that ever reads it is unwired dead code — so every clip move/trim/stretch/delete shortcut runs identically in both profiles regardless of which one is active.

In the flat profile only, <Shortcut client:load keys="f6" /> / <Shortcut client:load keys="shift+f6" /> cycles focus between major UI blocks (File → Home → Play → Add Track → track controls) as a fallback way to move between regions now that arrow-key grouping is off.

One footgun to know about: profiles are resolved by id, and passing an id that isn't `au4-tab-groups` or `wcag-flat` doesn't error and doesn't fall back to the default AU4 profile — it silently returns the <strong>flat</strong> profile instead. Anything that needs the tab-group behaviour described on this page must pass the real `au4-tab-groups` id explicitly.
</Callout>

## Implementation notes

- The group configuration (which groups exist, whether arrows/wrap are on) and the tab order table above are read from a single profile data structure, not scattered per-component constants.
- The tab order is fixed by configuration; it is not affected by DOM order within a group's container.
- Both the blur reset and the focus-on-entry reset matter in practice: without them, re-entering a group — a toolbar, a dialog's field group — would resume on whatever item was last focused instead of a predictable first item.
